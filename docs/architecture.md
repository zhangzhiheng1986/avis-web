## 1. 系统设计哲学

### 1.1 INCOSE SE 方法论基础

Avis 系统的设计严格遵循 INCOSE（国际系统工程委员会）系统工程方法论，核心原则包括：

- **全生命周期视角**：覆盖概念→开发→生产→使用→退役的知识全链条
- **需求驱动**：从利益相关者需求出发，逐层分解至系统需求、子系统规格
- **Vee 模型映射**：手册五编结构对应 Vee 模型的分解→集成双翼
- **可追溯性**：版本演化、交叉引用、引证金字塔构成完整的追溯链路
- **迭代增量**：版本演化路线体现系统工程迭代增量特征

### 1.2 设计目标

| 目标 | 描述 | 度量 |
|------|------|------|
| **知识持久化** | 阀门领域知识的结构化存储与版本控制 | Git 完整版本历史 |
| **可演化性** | 支持知识体系的持续迭代与增量更新 | 语义化版本机制 |
| **可协作性** | 多角色协同贡献的标准化流程 | RFC 提案-评审-合并 |
| **可检索性** | 多维度知识检索与交叉引用 | 知识图谱 + 全文索引 |
| **可发布性** | 多格式输出与Web发布能力 | Markdown → PDF/HTML/静态站点 |

## 2. 四层架构

```
┌──────────────────────────────────────────────────────────────┐
│                       发布层 (Publish)                        │
│               web/site/ — 静态站点生成（预留）                 │
├──────────────────────────────────────────────────────────────┤
│                       工具层 (Tools)                         │
│      tools/scripts/ — 统计、构建、质量检查                     │
│      tools/configs/ — lint规则、格式规范                      │
├──────────────────────────────────────────────────────────────┤
│                      协作层 (Collaboration)                   │
│   collaboration/templates/ — 内容模板                          │
│   collaboration/proposals/ — RFC提案                          │
│   collaboration/reviewers/ — 同行评审记录                      │
├──────────────────────────────────────────────────────────────┤
│                      知识层 (Knowledge)                       │
│   core/V5.0/ — 当前版本快照（Part1-6 + Appendix）              │
│   core/archive/ — 历史版本归档                                │
│   knowledge-graph/ — 本体、术语、交叉引用                      │
└──────────────────────────────────────────────────────────────┘
```

### 2.1 知识层（Knowledge Layer）

知识层是系统的核心资产层，包含手册的完整内容与结构化组织。

**目录结构**：
```
core/
├── V5.0/                          # 当前版本
│   ├── Part1-绪论与基础科学/       # 第 1-5 章
│   ├── Part2-设计工程/            # 第 6-26 章
│   ├── Part3-制造工艺与试验验证/    # 第 29-36 章
│   ├── Part4-系统工程与管理/       # 第 37-52 章
│   ├── Part5-前沿技术与未来发展/   # 第 53-60 章
│   ├── Part6-扩展/                # 第 61-62 章
│   └── Appendix/                  # 附录 A-E
└── archive/                       # 历史版本归档
    ├── V0.1/
    ├── V1.0/
    ├── V2.0/
    ├── V3.0/
    └── V4.0/
```

### 2.2 图谱层（Knowledge Graph Layer）

知识图谱为 Avis 系统提供语义化组织能力，包含三个子模块：

| 子模块 | 目录 | 内容 |
|--------|------|------|
| **本体定义** | `ontology/` | 阀门分类学、材料体系分类、故障模式分类、驱动方式分类等本体模型 |
| **术语词典** | `glossary/` | 航空航天阀门领域标准术语定义，含中英文对照、缩写全称 |
| **交叉引用** | `cross-references/` | 跨章引用索引、公式引用索引、标准引用索引 |

### 2.3 协作层（Collaboration Layer）

协作层使 Avis 从个人知识库升级为共创平台。

**核心流程**：提案 → 评审 → 合并 → 发布

| 阶段 | 产出物 | 存放位置 |
|------|--------|----------|
| 提案 | RFC 文档 | `collaboration/proposals/` |
| 评审 | 评审记录 | `collaboration/reviewers/` |
| 模板 | 标准化模板 | `collaboration/templates/` |

### 2.4 工具层（Tools Layer）

工具层提供自动化能力，保障知识库的质量与一致性。

| 工具类型 | 脚本 | 功能 |
|----------|------|------|
| 统计脚本 | `stats.py` | 字数统计、章节统计、图表统计 |
| 构建脚本 | `build.py` | Markdown → PDF/HTML 多格式输出 |
| 质量检查 | `lint.py` | 交叉引用校验、格式一致性检查、死链检测 |

### 2.5 发布层（Publish Layer）

发布层当前为预留设计，未来将支持：
- 静态站点生成（Hugo/Jekyll）
- PDF 出版物编译（LaTeX 模板）
- 在线知识库网站

## 3. 版本管理策略

### 3.1 语义化版本

采用 `MAJOR.MINOR.PATCH` 语义化版本规范：

| 版本位 | 变更条件 | 示例 |
|--------|----------|------|
| MAJOR | 知识体系结构重大调整，编章重组 | V5.0 → V6.0 |
| MINOR | 新增章节、重大内容扩充 | V5.0 → V5.1 |
| PATCH | 勘误修正、格式优化、引用更新 | V5.1 → V5.1.1 |

### 3.2 Git 分支策略

```
main            ← 稳定发布分支（只接受已评审的合并）
├── develop     ← 开发分支
│   ├── feat/*  ← 功能/内容分支
│   ├── fix/*   ← 修正分支
│   └── rfc/*   ← 提案分支
```

### 3.3 版本归档机制

- 每个 MAJOR 或 MINOR 版本发布后，将完整快照归档至 `core/archive/<version>/`
- 归档时保留完整的文件结构和版本元数据
- 历史版本仅供追溯，不得修改

## 4. 知识图谱设计

### 4.1 本体模型（Ontology）

核心本体类别：

```
阀门分类学 (Valve Taxonomy)
├── 按驱动方式: 电磁 | 气动 | 液压 | 电机 | 火工/电爆 | 压电/智能材料
├── 按功能类型: 单向阀 | 泄压阀 | 截止阀 | 调节阀 | 球阀 | 蝶阀
├── 按应用场景: 推进剂阀 | 燃料阀 | 氧化剂阀 | 吹除阀 | 排气阀
└── 按工作环境: 低温阀 | 高温阀 | 高压阀 | 真空阀 | 腐蚀介质阀

材料体系 (Material System)
├── 金属材料: 钛合金 | 镍基合金 | 不锈钢 | 铝合金
├── 非金属材料: 密封材料 | 润滑材料 | 涂层材料
└── 复合材料: C/C | C/SiC | 金属基复合材料

故障模式 (Failure Mode)
├── 泄漏模式
├── 卡滞模式
├── 电磁故障
└── 疲劳/断裂
```

### 4.2 交叉引用系统

五级引证金字塔：
- **A 级**：权威标准（ISO、MIL、国军标等）
- **B 级**：核心教材与专著
- **C 级**：学术论文与会议论文
- **D 级**：工业报告与技术手册
- **E 级**：工程经验与专家判断

## 5. 共创流程

### 5.1 RFC 提案流程

```
1. 发现问题/改进点
   ↓
2. 撰写 RFC 提案文档（使用 templates/rfc-template.md）
   ↓
3. 提交至 collaboration/proposals/
   ↓
4. 同行评审（至少 1 位领域专家）
   ↓
5. 修改与迭代
   ↓
6. 评审通过，合并至主分支
   ↓
7. 版本发布，更新 CHANGELOG
```

### 5.2 贡献者角色

| 角色 | 权限 | 职责 |
|------|------|------|
| **维护者** (Maintainer) | 合并权限 | 最终审核、版本发布 |
| **评审者** (Reviewer) | 评审权限 | 技术评审、质量把关 |
| **贡献者** (Contributor) | 提案权限 | 提交 RFC、内容编写 |

## 6. 质量保证体系

### 6.1 内容质量标准

- 所有新增内容必须有引用来源（至少 C 级以上）
- 交叉引用必须经过自动化校验
- 公式格式统一使用 LaTeX 语法
- 表格必须有标题和编号

### 6.2 自动化检查

| 检查项 | 工具 | 触发条件 |
|--------|------|----------|
| 交叉引用完整性 | `lint.py` | 每次提交 |
| 格式一致性 | `lint.py` | 每次提交 |
| 死链检测 | `lint.py` | 每周 |
| 字数统计 | `stats.py` | 版本发布前 |

## 7. 技术栈选型

| 层面 | 技术 | 选型理由 |
|------|------|----------|
| **版本控制** | Git | 业界标准，分布式，完整历史 |
| **文档格式** | Markdown (GFM) | 纯文本，版本可控，多格式输出 |
| **公式** | LaTeX | 数学公式标准，渲染成熟 |
| **自动化** | Python 3.11+ | 脚本生态丰富，跨平台 |
| **构建** | Pandoc + Make | 多格式文档转换 |
| **Web（预留）** | Hugo + GitHub Pages | 静态站点，Markdown 原生支持 |
| **图谱（预留）** | Neo4j / RDF | 知识图谱存储与查询 |
