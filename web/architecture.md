## 一、系统概述

### 1.1 系统标识

| 属性 | 内容 |
|------|------|

| **中文名称** | 航空航天阀门研发智库 |

| **系统定位** | 全球航空航天阀门领域知识体系的 **活的大厦** — 可版本演化、支持共创的领域知识基础设施 |

### 1.3 使命声明

> Avis 系统的使命是：以系统工程方法论为指导，将航空航天阀门领域的分散知识整合为结构化、可演化、可协作的开放知识基础设施，为全球阀门工程师、研究人员和学者提供权威、完整、持续更新的知识服务体系。

### 1.4 系统边界定义

**系统包含**：

- 航空航天阀门研发设计手册全部内容（5大编 57章 + 5附录，约60万汉字）
- 知识体系的结构化组织（本体定义、术语词典、交叉引用索引）
- 版本管理与演化机制（Git 版本控制 + 语义化版本 + 全量 CHANGELOG）
- Web 站点及全文搜索（VitePress 静态站点 + MiniSearch 本地搜索）
- CI/CD 自动构建与部署流水线（GitHub Actions + GitHub Pages）
- 共创协作平台（RFC 提案流程、评审机制、五级角色权限）

**系统不包含**：

- 实时协作编辑功能（V7.0 路线图中）
- AI 智能问答与检索增强生成（Phase 4 路线图中）
- 数字孪生集成与仿真工具对接（V7.0+ 路线图中）
- 多语言翻译版本（中期路线图中）
- 商业出版与印刷服务

### 1.5 核心价值主张

| 价值维度 | 内涵 | 实现机制 |
|----------|------|----------|
| **知识持久化** | 阀门领域知识的结构化存储与完整历史追溯 | Git 版本库 + `core/archive/` 版本归档 |
| **可演化性** | 支持知识体系的持续迭代与增量更新 | 语义化版本规范 + MAJOR/MINOR/PATCH 三级发布机制 |
| **可协作性** | 多角色协同贡献的标准化流程 | RFC 提案-评审-合并-发布四阶段流水线 |
| **可检索性** | 多维度知识检索与交叉引用 | 知识图谱本体化组织 + VitePress 全文搜索 |
| **可发布性** | 多格式输出与全球 Web 可访问 | VitePress 静态站点 + GitHub Pages 全球 CDN |
| **工程严谨性** | 基于系统工程方法论的知识质量控制 | INCOSE SE 方法论 + 五级引证金字塔 + 自动化 lint |

## 二、系统架构全景图

### 2.1 设计哲学：INCOSE SE 方法论驱动

Avis 系统的架构设计严格遵循 INCOSE（国际系统工程委员会）系统工程方法论，将手册的知识体系构建和系统平台的技术实现统一在同一个系统工程框架下。

**核心设计原则**：

| 原则 | 来源 | 在 Avis 中的体现 |
|------|------|-----------------|
| 全生命周期视角 | INCOSE SE 技术过程 | 覆盖概念→设计→制造→验证→运维的阀门知识全链条 |
| 需求驱动分解 | INCOSE 利益相关者需求定义 | 从目标读者画像出发，逐层分解至五编57章的知识需求 |
| Vee 模型映射 | INCOSE Vee 模型 | 手册五编对应 Vee 模型的分解→集成双翼 |
| 可追溯性 | INCOSE 配置管理 | 版本演化 + 交叉引用 + 五级引证金字塔 |
| 迭代增量 | INCOSE 迭代开发原则 | V0.1→V5.0 的渐进式知识积累 |

### 2.4 各层详细职责

#### 2.4.2 图谱子层（Knowledge Graph Sub-Layer）

图谱子层为知识层提供语义化组织能力，是连接知识内容与用户检索的关键中间层。

| 模块 | 路径 | 职责 |
|------|------|------|
| 本体定义 | `knowledge-graph/ontology/` | 阀门分类学、材料体系、故障模式、驱动方式等本体模型 |
| 术语词典 | `knowledge-graph/glossary/` | 航空航天阀门领域标准术语定义（中英文对照、缩写全称） |
| 交叉引用 | `knowledge-graph/cross-references/` | 跨章引用索引、公式引用索引、标准引用索引 |

#### 2.4.3 协作层（Collaboration Layer）

协作层使 Avis 从个人知识库升级为共创平台。

| 模块 | 路径 | 职责 |
|------|------|------|
| 贡献指南 | `collaboration/CONTRIBUTING.md` | 贡献流程说明、角色定义、评审标准 |
| RFC 提案 | `collaboration/proposals/` | 改进提案文档（结构化 RFC 格式） |
| 评审记录 | `collaboration/reviewers/` | 同行评审过程记录 |
| 内容模板 | `collaboration/templates/` | 新章节模板、评审模板、勘误模板（3个模板文件） |

核心流程：`提案 → 讨论（≥2周）→ 核心团队评审 → PR 合并 → 版本发布`

#### 2.4.4 工具层（Tool Layer）

工具层提供自动化能力，保障知识库的质量与一致性。

| 工具 | 路径 | 功能 | 状态 |
|------|------|------|------|
| 部署脚本 | `tools/scripts/deploy.ps1` | 本地构建 + 部署到 GitHub Pages | 已就绪 |
| 统计脚本 | `tools/scripts/stats.py`（规划中） | 字数统计、章节统计、图表统计 | V5.2 |
| 质量检查 | `tools/scripts/lint.py`（规划中） | 交叉引用校验、死链检测、格式检查 | V5.2 |
| 构建配置 | `tools/configs/` | lint 规则、格式规范 | 预留 |

#### 2.4.5 发布层（Publication Layer）

发布层将知识层的 Markdown 源文件转化为全球可访问的 Web 站点。

| 组件 | 技术选型 | 职责 | 状态 |
|------|----------|------|------|
| 静态站点 | VitePress 1.6.4 | 构建 + 全文搜索 + 数学渲染 | 已上线 |
| 托管平台 | GitHub Pages | 静态资源托管 | 已部署 |
| CDN 加速 | Fastly（GH Pages 内置） | 全球内容分发 | 自动启用 |
| CI/CD | GitHub Actions | 自动化构建与部署 | workflow 就绪 |

## 三、技术栈全景

### 3.1 技术选型总表

| **公式渲染** | LaTeX (MathJax 3) | 3.x | 数学公式标准，VitePress 原生集成 |
| **静态站点生成** | VitePress | 1.6.4 | 基于 Vue 3 + Vite，构建速度极快，内置本地搜索 |
| **前端框架** | Vue 3 | 3.x | VitePress 底层框架，支持组件化扩展 |
| **构建工具** | Vite | 6.x | 新一代前端构建工具，HMR 极速 |
| **搜索** | MiniSearch（VitePress 内置） | — | 纯前端本地搜索，零配置，中文分词支持 |
| **数学渲染** | markdown-it-mathjax3 | — | Markdown 内嵌 LaTeX 渲染插件 |
| **语法高亮** | Shiki | — | VitePress 内置，支持数百种语言 |
| **部署** | GitHub Actions + peaceiris/actions-gh-pages@v3 | — | 自动化 CI/CD，免费，稳定 |
| **托管** | GitHub Pages | — | 免费 100GB/月带宽，Fastly 全球 CDN |
| **运行时** | Node.js | 18 | LTS 版本，稳定可靠 |
| **包管理** | npm | — | Node.js 标准包管理器 |
| **协作** | GitHub Issues / PR | — | 问题追踪 + 代码审查 + 社区讨论 |
| **评论系统** | Giscus（预留） | — | 基于 GitHub Discussions，无服务器，免费 |

### 3.2 VitePress 选型对比分析

| 维度 | VitePress | Docusaurus | Hugo | MkDocs |
|------|-----------|------------|------|--------|
| 构建速度 | **极快（Vite 驱动）** | 较快（Webpack） | 极快（Go 编译） | 快 |
| Markdown 支持 | **原生 + Vue 组件** | 原生 + React | 原生 | 原生 |
| 全文搜索 | **内置（MiniSearch）** | 需 Algolia（付费） | 无内置 | 需插件 |
| 数学公式 | **MathJax 3 原生集成** | 需配置 | 需配置 | 需插件 |
| 中文生态 | **优秀（Vue 官方文档用）** | 良好 | 一般 | 良好 |
| 自定义主题 | **简单（Vue 组件）** | 较复杂（React） | 复杂（Go 模板） | 有限 |
| 学习曲线 | **低** | 中 | 低 | 低 |

VitePress 在中文技术文档领域有广泛使用（Vue.js 官方文档、Vite 官方文档均使用 VitePress），构建速度极快（测试构建 14.05s），内置全文搜索无需第三方服务，Vue 3 组件支持为后续知识图谱等交互功能预留扩展基础。

## 六、版本管理策略

### 6.1 语义化版本规范（SemVer）

Avis 系统严格遵循 `MAJOR.MINOR.PATCH` 语义化版本规范：

| 版本位 | 变更条件 | 典型变更 | 发布节奏 |
|--------|----------|----------|----------|
| **MAJOR** | 知识体系结构重大调整 | 编章重组、分类体系变更、平台架构升级 | 每半年至一年 |
| **MINOR** | 新增章节或重大内容扩充 | 新增 ≥1 章，或新增 ≥5000 字内容 | 每月至每季度 |
| **PATCH** | 勘误修正与小幅改进 | 错别字修正、格式优化、引用更新、小幅内容增补（<5000字） | 按需，每周可能多次 |

### 6.2 Git 分支策略

```
master (稳定发布分支 — 只接受已评审的合并)
  │
  ├── gh-pages (自动部署分支 — GitHub Actions 自动生成)
  │
  └── 特性分支 (短期)
      ├── feat/chapter-xxx    ← 内容新增分支
      ├── fix/typo-xxx        ← 修正分支
      └── rfc/rfc-xxxx        ← 提案分支
```

采用 **Trunk-Based Development** 模式：
- `master` 为唯一长期分支，始终保持可发布状态
- 所有变更通过短期特性分支 + PR 合并
- `gh-pages` 由 CI/CD 自动维护，禁止手动操作
- 每个版本对应一个 Git Tag（如 `v5.0`、`v5.1`）

### 7.2 工作流配置详解

```yaml
# .github/workflows/deploy.yml

name: Deploy VitePress Site to GitHub Pages

on:
  push:
    branches:
      - master                          # 仅 master 分支 push 触发

concurrency:
  group: pages                          # 防止并发部署冲突
  cancel-in-progress: true

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest              # GitHub 托管运行器

    steps:
      - name: Checkout
        uses: actions/checkout@v4       # 检出源码

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: 18              # Node.js 18 LTS

      - name: Install dependencies
        run: |
          cd web
          npm ci                        # 严格安装（非 npm install）

      - name: Build VitePress
        run: |
          cd web
          npx vitepress build
        env:
          NODE_OPTIONS: --max-old-space-size=4096   # 分配 4GB 内存
          BASE: /${{ github.event.repository.name }}/ # 自动适配仓库名

      - name: Deploy to GitHub Pages
        uses: peaceiris/actions-gh-pages@v3
        with:
          github_token: ${{ secrets.GITHUB_TOKEN }}
          publish_dir: web/.vitepress/dist
          publish_branch: gh-pages       # 目标分支
          force_orphan: true             # 强制孤立提交（保持分支干净）
```

### 7.3 自动化触发条件

| 触发事件 | 触发条件 | 说明 |
|----------|----------|------|
| **push** | push 到 `master` 分支 | 内容更新后自动构建部署 |
| **并发控制** | 同一 `pages` 组内互斥 | 防止多个部署同时运行导致覆盖 |
| **手动触发** | workflow_dispatch（待添加） | 支持手动重新部署 |

| 验证项 | 方法 | 预期 |
|--------|------|------|
| 构建成功 | Vitepress build 返回码 = 0 | 无错误信息 |

## 八、共创协作机制

### 8.1 五级贡献者角色模型

| 角色 | 权限范围 | 获取方式 | 职责描述 |
|------|----------|----------|----------|
| **浏览者（Visitor）** | 只读访问 | 默认 | 浏览学习、通过 Issue 反馈错误或建议 |
| **贡献者（Contributor）** | Fork → PR | 首次成功提交 RFC 或 PR | 提交改进提案、修复错误、补充内容 |
| **评审者（Reviewer）** | PR Review | 持续高质量贡献 + 邀请 | 审核技术内容准确性、格式规范性 |
| **维护者（Maintainer）** | Merge 权限 | 核心贡献 + 任命 | 合并 PR、发布版本、管理 Issue |
| **作者（Author）** | 所有权限 | 创始者 | 最终决策、版本规划、质量把关 |

角色晋升路径：浏览者 →（首次贡献）→ 贡献者 →（持续高质量贡献）→ 评审者 →（核心贡献 + 任命）→ 维护者

### 8.2 RFC 提案流程

```
┌──────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────┐
│  发现问题  │───→│  创建 RFC     │───→│  社区讨论     │───→│  核心团队评审  │───→│  PR 合并  │
│  或改进点  │    │  提案 Issue   │    │  (≥2周)       │    │              │    │  版本发布  │
└──────────┘    └──────────────┘    └──────────────┘    └──────────────┘    └──────────┘
```

**RFC 适用场景（必须走 RFC 流程）**：
- 新增章节或重大内容扩充（>500字新增内容）
- 修改现有章节的核心结论或技术判断
- 调整知识体系结构（编章重组）
- 修改术语定义或分类体系

**RFC 豁免场景（可直接提交修正）**：
- 错别字、标点修正
- 格式统一化调整
- 引用编号修正
- 死链修复

**RFC 文档要素**：
- 提案标题（简洁描述改进内容）
- 动机与背景（为什么要改）
- 详细方案（具体怎么改）
- 影响范围（涉及哪些章节）
- 参考依据（引用的标准、文献，至少 C 级以上）
- 提案人信息

### 8.3 评审标准

| 维度 | 标准 | 权重 |
|------|------|------|
| **技术准确性** | 内容是否符合工程科学原理，有权威引用支撑 | 40% |
| **体系一致性** | 是否与现有知识体系自洽，不引入矛盾 | 30% |
| **覆盖完整性** | 是否填补了已知的知识空白 | 20% |
| **表达清晰性** | 表述是否清晰、逻辑是否严密 | 10% |

**表格规范**：
- 表格必须有序号和标题：`表 X.Y 表格标题`
- 使用标准 Markdown 表格语法
