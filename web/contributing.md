# 贡献指南

> 欢迎加入Avis知识体系的共建！本指南将帮助你了解如何参与贡献。

---

## 为什么参与贡献？

Avis是全球航空航天阀门领域知识体系的"活的大厦"。你的每一次贡献——无论是修正一个错别字、补充一段技术细节，还是撰写一个全新章节——都将让这座大厦更加坚固、完备。

---

## 贡献者角色与权限

| 角色 | 参与方式 | 典型贡献 |
|------|----------|----------|
| 浏览者 | GitHub Issue | 报告错误、提出改进建议 |
| 贡献者 | Fork → Pull Request | 修正内容、补充章节、翻译 |
| 评审者 | PR Review | 审核技术准确性、格式规范性 |
| 维护者 | Merge + Release | 合并PR、发布版本 |
| 作者 | 最终决策 | 版本规划、架构设计 |

## RFC提案流程

Avis采用RFC（Request for Comments）流程管理重大变更：

```
1. 提出提案 → 2. 社区讨论 → 3. 核心评审 → 4. 实施合并 → 5. 版本发布
   (2周讨论期)    (公开讨论)     (≥2位评审)    (PR通过)      (Git Tag)
```

### 第一步：创建RFC提案

在 `collaboration/proposals/` 目录下创建提案文件，使用 [RFC模板](https://github.com/avis-aerospace/avis-web/blob/main/collaboration/templates/rfc-template.md)，包含：

- **提案标题**：简洁描述改进内容（如 "RFC: 新增第XX章 低温阀门热力学分析"）
- **动机与背景**：为什么要做这个改进？解决什么问题？
- **详细方案**：具体怎么改？涉及哪些章节？
- **影响范围**：对现有内容的影响评估
- **参考依据**：引用的标准、文献、工程案例

### 第二步：社区讨论

- 提案提交后，在GitHub Discussions中进行公开讨论
- 讨论周期至少2周，确保充分的社区反馈
- 提案人需积极回应讨论中的问题

### 第三步：核心评审

- 由核心团队（≥2位评审者）进行技术评审
- 评审标准：技术准确性、逻辑一致性、格式规范性、引用完备性

### 第四步：实施与合并

- 评审通过后，贡献者提交Pull Request
- PR需通过自动化检查（Markdown语法、链接有效性）
- 维护者确认后合并

### 第五步：版本发布

- 合并后按版本编号规则更新版本号
- 更新CHANGELOG.md
- 打Git Tag，触发自动部署

## 内容编写规范

### Markdown标准

- 使用 `.md` 扩展名，UTF-8编码
- 章节标题使用 `#` 到 `####` 四级标题
- 表格使用标准Markdown表格语法
- 代码块指定语言：````matlab`、````python` 等

### 引用格式

- 参考文献使用 `[编号]` 标注，在章末列出完整信息
- 交叉引用使用 `[参见第X章]` 格式
- 标准引用标注标准编号和版本年份

### 公式规范

- 行内公式使用 `$...$`
- 块级公式使用 `$$...$$`
- 变量使用斜体，单位使用正体
- 向量使用粗体 `\mathbf{}`

### 图片规范

- 图片存放在 `images/` 目录
- 使用相对路径引用
- 建议分辨率：宽度800-1200px
- 格式：PNG（截图）、SVG（示意图）

## 评审标准

| 评审维度 | 标准 |
|----------|------|
| 技术准确性 | 公式推导正确、数据有据可查、结论逻辑严谨 |
| 内容完整性 | 覆盖章节目标的所有关键点 |
| 结构清晰度 | 层次分明、逻辑递进、便于导航 |
| 引用完备性 | 关键论断有参考文献支撑 |
| 术语一致性 | 与全书术语体系一致 |
| 格式规范性 | 符合Markdown编写规范 |

---

## 版本编号规则

遵循语义化版本 SemVer 2.0：`主版本.次版本.修订号`

- **修订号**（V5.0.x）：勘误修正、小改进、格式优化
- **次版本**（V5.x.0）：新增章节、重大修订、功能升级
- **主版本**（Vx.0.0）：知识体系重构、平台架构升级
